⚡ What is HLD?

High Level Design focuses on the architecture of a large-scale distributed system — scalability, availability, reliability, and trade-offs. Think: "Design Twitter", "Design Netflix", "Design URL Shortener".

🧠
Core Concepts to Master
Non-negotiable fundamentals
Scalability Tier 1
  • Horizontal vs Vertical scaling
  • Load balancing (Round Robin, Least Conn)
  • Stateless vs Stateful services
  • Auto-scaling strategies
Databases Tier 1
  • SQL vs NoSQL trade-offs
  • Sharding & Partitioning
  • Read replicas & replication
  • CAP theorem in practice
Caching Tier 1
  • Redis / Memcached
  • Cache-aside, Write-through, Write-back
  • Cache invalidation strategies
  • CDN for static content
Messaging & Queues Tier 1
  • Kafka, RabbitMQ, SQS
  • Pub/Sub vs message queues
  • Event-driven architecture
  • At-least / exactly-once delivery
Networking Tier 2
  • DNS, HTTP/HTTPS, TCP/UDP
  • REST vs GraphQL vs gRPC
  • WebSockets, Long polling, SSE
  • API Gateway patterns
Reliability Tier 2
  • Rate limiting (Token bucket, Leaky bucket)
  • Circuit breaker pattern
  • Idempotency & retries
  • Distributed transactions (Saga)

The HLD Interview Framework
Follow this structure every time
STEP 1 — 5 min
Clarify Requirements
Functional requirements (what it does), Non-functional (scale, latency, availability, consistency). Ask: DAU, read/write ratio, data size, SLA.
STEP 2 — 5 min
Capacity Estimation
Back-of-envelope: QPS, storage, bandwidth. E.g. 100M DAU × 10 requests/day = ~11,500 QPS. Show your math — interviewers love this.
STEP 3 — 5 min
High-Level Architecture
Draw: Client → API Gateway → Services → DB/Cache. Identify core components. Define API endpoints briefly.
STEP 4 — 15 min
Deep Dive Components
Pick 2–3 tricky parts (DB schema, caching layer, message queue, consistency). Go deep. Discuss trade-offs proactively.
STEP 5 — 5 min
Bottlenecks & Scale
Where does your system break at 10×? Discuss sharding, horizontal scaling, failover, monitoring, alerting.

⚡ What is LLD?

Low Level Design focuses on class design, OOP principles, and design patterns. Think: "Design a Parking Lot", "Design Chess Game", "Design an Elevator System". It's about code structure, not infrastructure.

🧱
OOP Principles — Nail These First
The foundation of every LLD answer
SOLID Principles Must Know
  • Single Responsibility — one class, one job
  • Open/Closed — open to extend, closed to modify
  • Liskov Substitution — subclass is substitutable
  • Interface Segregation — small, focused interfaces
  • Dependency Inversion — depend on abstractions
OOP Pillars Must Know
  • Encapsulation — hide internal state
  • Abstraction — expose only what's needed
  • Inheritance — reuse via is-a relationships
  • Polymorphism — one interface, many forms
🎨
Design Patterns to Know
GoF patterns that actually appear in interviews
Creational High Frequency
  • Singleton — Logger, Config, DB Connection pool
  • Factory / Abstract Factory — Payment processors, Notification types
  • Builder — Complex objects like Query builders, HTTP requests
Structural High Frequency
  • Adapter — Wrap incompatible interfaces
  • Decorator — Pizza toppings, I/O streams, middleware
  • Facade — Simplify complex subsystems
  • Composite — File system tree, UI components
Behavioral High Frequency
  • Observer — Event systems, notifications, pub/sub
  • Strategy — Sorting algorithms, payment methods
  • Command — Undo/redo, task queues
  • State — Vending machine, traffic lights, order status
  • Template Method — Game loop, data processing pipelines
📐
LLD Interview Framework
The right approach in the interview room
STEP 1 — 3 min
Gather Requirements
Functional: what actors exist? What actions can they perform? Non-functional: scale, concurrency, extensibility needs.
STEP 2 — 5 min
Identify Entities & Relationships
List nouns = potential classes. List verbs = potential methods. Define is-a (inheritance) vs has-a (composition) relationships.
STEP 3 — 5 min
Design Class Diagram
Draw classes with attributes, methods, access modifiers. Show interfaces/abstract classes. Show associations and multiplicities.
STEP 4 — 15 min
Code the Core Classes
Write actual code for 2–3 key classes. Apply design patterns where natural. Show error handling and edge cases.
STEP 5 — 5 min
Walk Through a Use Case
Trace a complete user action through your design. Discuss extensibility: "If we add feature X, we only need to change Y."

⚡ The Right LeetCode Mindset

Don't grind randomly. Learn patterns, not solutions. One pattern unlocks dozens of problems. Quality > Quantity. 150 well-understood problems beats 500 memorized ones.

📅
12-Week Structured Plan
2–3 hours/day, 5 days/week
WEEKS 1–2
Arrays, Strings & Hashing
Two pointers, sliding window, prefix sums, hashmaps, frequency counts. Master: Two Sum, Valid Anagram, Longest Substring Without Repeat, Product Except Self.
WEEKS 3–4
Stacks, Queues & Linked Lists
Monotonic stack, deque tricks, fast/slow pointers, merge/reverse. Master: Valid Parentheses, Daily Temperatures, LRU Cache, Merge Two Lists.
WEEKS 5–6
Trees & Binary Search
DFS/BFS on trees, BST properties, binary search variants, level-order traversal. Master: LCA, Kth Largest, Validate BST, Construct from Preorder.
WEEKS 7–8
Graphs & Backtracking
BFS/DFS traversal, Union-Find, Topological sort, Dijkstra. Combinations/permutations via backtracking. Master: Number of Islands, Course Schedule, Word Search, Subsets.
WEEKS 9–10
Dynamic Programming
1D DP → 2D DP → knapsack variants → interval DP. Master: Climbing Stairs, Coin Change, Longest Common Subsequence, Edit Distance, House Robber.
WEEKS 11–12
Heaps, Tries & Advanced
Priority queues, top-K problems, trie construction, segment trees. Mock interviews daily. Master: Merge K Lists, Find Median, Word Search II.

Problem-Solving Protocol
Do this for EVERY problem
BEFORE CODING
Read → Examples → Edge Cases
Read the problem twice. Trace through 2 examples by hand. Ask: what if input is empty? Negative numbers? Duplicates?
THINK ALOUD
Brute Force → Optimize
State the brute force O(n²) first. Then identify the bottleneck. Ask: can I use a hashmap? Sorting? Sliding window? Binary search?
CODING
Clean Code, Talk as You Type
Use meaningful variable names. Write helper functions. Comment non-obvious logic. Don't go silent.
AFTER CODING
Trace → Complexity → Test
Manually trace your code with an example. State time and space complexity. Test edge cases out loud.
